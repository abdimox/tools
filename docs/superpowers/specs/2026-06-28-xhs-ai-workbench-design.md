# 乐活互动小红书 AI 运营工作台设计规格

## 目标与范围

交付一个供乐活互动内部使用的一次性 AI 内容生成工作台。首版采用完整演示模式，不依赖外部 AI Key，但保留可替换的 AI Provider 边界。系统不使用数据库、不保存历史记录，不包含发布、爬取、私信、支付或复杂账号体系。

## 用户流程

用户通过简单访问密码登录后进入首页，可选择三个任务：生成小红书笔记、分析自己的账号、拆解同行爆款。所有任务只在当前页面保留结果；刷新后清空。

生成笔记时，用户选择“手作 DIY”或“Photobooth”，输入一句案例简介并上传最多 12 张活动图片。系统生成案例解析、图片建议、30 个标题、5 版正文、20 条封面文案、8–12 个标签、封面提示词及合规结果。用户可以复制任意文字结果，并基于上传图片生成、预览和下载 3:4 演示封面。

账号分析与同行分析均支持多图上传和补充文字。演示引擎输出结构化、可执行的分析内容，并严格区分两类业务。

## 架构

- 前端：React、TypeScript、Vite、React Router 与原生 CSS。
- 后端：Node.js、Express、TypeScript、Multer 与 Sharp。
- AI 层：统一 Provider 接口；首版实现本地 Demo Provider，后续真实 Provider 可在不改路由与前端数据结构的情况下接入。
- 鉴权：后端校验 `APP_PASSWORD`，使用后端密钥签发有时效的 HMAC Bearer Token。
- 文件：`temp-uploads` 与 `temp-outputs` 保存临时文件；上传文件在处理后尽快删除，输出文件默认 30 分钟清理，服务启动时清空临时目录。

## 前端结构

桌面端使用左侧导航和右侧工作区；移动端改为顶部导航和单列卡片。页面包含 Login、Dashboard、NoteGenerator、AccountAnalyzer 与 CompetitorAnalyzer。共享组件负责上传、复制、结果分区、加载反馈、错误提示与合规状态。

视觉采用专业、克制的橙白色运营后台风格；通过间距、排版和浅色分区建立层级，避免复杂动画、花哨背景与过度卡片嵌套。

## 后端接口与数据流

- `POST /api/auth/login`：密码换取短期访问令牌。
- `POST /api/generate-note`：接收业务类型、案例简述和图片，返回完整笔记包。
- `POST /api/generate-cover-image`：基于上传图片与封面文案生成 3:4 演示封面。
- `POST /api/compliance-check`：扫描风险词并给出原因和合规改写。
- `POST /api/analyze-account`：输出账号诊断与 14 天行动计划。
- `POST /api/analyze-competitor`：输出爆款结构、可借鉴方向及双业务改写建议。

前端以 `multipart/form-data` 上传图片；后端验证格式、数量和大小后调用 Demo Provider。路由只依赖 Provider 接口，不耦合具体实现。

## 演示模式行为

演示引擎从案例简述中提取城市、人数、客户类型、项目与场景，并结合业务默认服务和卖点生成真实可用的示例结果。所有内容数量满足执行文档要求。图片分析以文件顺序、尺寸和方向生成封面推荐、排序、裁剪与提亮建议。

封面生成使用 Sharp 对首选上传图进行 3:4 裁剪、适度提亮并叠加标题，生成可下载的 PNG。此能力用于演示完整工作流，不宣称为生成式 AI。

## 合规与错误处理

合规服务扫描标题、正文、封面文案、标签与提示词中的私信、微信、VX、二维码、报价、扫码等风险表达，返回风险位置、原因和安全改写。业务类型、简介、上传数量、格式和大小均由前后端双重校验。

未登录返回 401；输入错误返回 400；上传错误返回明确中文提示；服务异常返回 500 且不泄露环境变量或内部路径。前端在每次异步操作中显示加载状态，并提供可恢复的错误反馈。

## 测试与验收

- 单元测试：案例解析、业务隔离、内容数量、合规扫描、令牌签发与校验。
- 接口测试：登录、受保护路由、笔记生成、账号分析、同行分析和封面生成。
- 构建验证：前后端 TypeScript 检查与 Vite 生产构建。
- 浏览器验收：桌面和手机视口检查登录、导航、上传、生成、复制、下载、加载及错误状态。

## 成功标准

新环境按 README 配置密码并启动后，不配置任何 AI Key 即可完整演示三个工作流；所有必需输出可查看和复制，封面可下载；刷新不保留历史；API Key 不进入前端；移动端和桌面端均可正常使用。
