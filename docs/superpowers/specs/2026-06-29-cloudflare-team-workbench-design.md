# 乐活互动 AI 运营工作台：Cloudflare 多员工版设计

## 目标

将当前本地运行的 React/Node 工具改造成可由少量内部员工长期在线使用的工作台，并通过 GitHub 自动部署到 Cloudflare 免费服务。每位员工使用独立密码登录，只能看到自己的 GPT 对话。生成笔记继续以提升小红书点击与咨询意愿为目标；封面图制作缩减为“粘贴提示词、上传图片、生成图片”三项操作。

本次改造优先保证：真实 AI 调用、数据隔离、内容质量、低使用门槛和免费部署。界面保持简单，不增加与当前业务无关的管理功能。

## 已确认的范围

### 包含

- GitHub 保存代码，Cloudflare 从 GitHub 自动部署。
- Cloudflare Pages 承载前端，Pages Functions（Workers 运行时）承载 API。
- Cloudflare D1 保存员工、会话、对话和消息。
- Cloudflare R2 保存聊天附件以及短期封面输入、输出图片。
- 管理员创建、停用员工账号和重置密码。
- 员工只输入自己的独立密码即可登录。
- 生成笔记严格输出 3 个标题和 1 篇带话题的正文，并在展示前完成二次审校。
- 在“生成笔记”下面新增 GPT 对话。
- GPT 对话支持新建、重命名、删除、自动保存、连续上下文和图片上传。
- 封面图页面仅包含提示词文本框、图片上传、生成按钮和结果保存。
- 继续使用两个独立 API Key：文字功能使用文字 Key，图片生成功能使用图片 Key。

### 不包含

- 公开注册、短信或邮箱验证、找回密码、计费和套餐。
- 员工之间共享或查看彼此对话。
- 模型选择器、插件、联网搜索、语音对话和文件知识库。
- 封面提示词生成、原图分析、评分、最佳图片推荐、封面文字候选和封面历史。
- 购买服务器、购买域名或部署数据库服务器。
- 在代码仓库中保存任何 API Key、密码或加密密钥。

## 信息架构

登录后的左侧导航顺序为：

1. 首页
2. 生成笔记
3. GPT 对话
4. 封面图制作
5. 账号分析
6. 同行分析
7. 设置

“设置”中的员工管理和接口配置只对管理员显示。普通员工可以使用业务功能，但不能读取或修改员工、模型、地址和 API Key。

## 部署架构

采用一个 GitHub 仓库和一个 Cloudflare Pages 项目：

- Vite 构建 React 静态资源。
- Pages Functions 提供 `/api/*` 路由。
- D1 通过绑定提供关系数据。
- R2 通过私有绑定提供图片存储。
- GitHub Actions 在主分支提交后先应用版本化 D1 migration，再构建并发布 Pages；拉取请求只生成不写正式 D1 的预览部署。
- 正式地址使用 Cloudflare 提供的 `pages.dev` 域名；自定义域名为可选项，不是上线前提。

现有前端页面和可复用业务逻辑保留。Express、Node 文件系统、Node 专用加密和本地配置文件改为 Workers 兼容实现。HTTP 路由层与业务服务层分离，避免将 Cloudflare 绑定散落到提示词和内容规则中。

## 账号与权限

### 密码登录

每个员工有一个显示名称和一个独立密码，登录页只显示密码输入框。后台通过带服务端 pepper 的不可逆查找指纹定位账号，再验证带独立盐值的慢哈希。数据库不保存明文密码。

用户表包含：

- `id`
- `display_name`
- `password_lookup`
- `password_hash`
- `password_salt`
- `role`（`admin` 或 `employee`）
- `active`
- `created_at`
- `updated_at`

`password_lookup` 必须唯一，避免两位员工设置相同密码而造成身份混淆。

### 会话

登录成功后创建随机会话令牌，只把令牌哈希和到期时间写入 D1。浏览器使用 `HttpOnly`、`Secure`、`SameSite=Lax` Cookie，不把认证令牌放入 localStorage。退出、停用账号和重置密码时使现有会话失效。

登录接口增加按来源和密码指纹的失败次数限制。错误信息统一为“密码错误或账号已停用”，不暴露账号是否存在。

### 管理员

首次部署通过 Cloudflare Secret 中的引导管理员密码创建第一个管理员。管理员创建成功后，不再依赖引导密码登录。管理员页面支持：

- 添加员工名称和初始密码。
- 修改员工名称。
- 重置员工密码。
- 启用或停用员工。
- 查看员工最后登录时间。

不提供员工对话查看入口。

## 数据模型与隔离

D1 增加以下核心表：

- `users`
- `sessions`
- `conversations`
- `messages`
- `message_attachments`
- `ai_config`

`conversations.owner_user_id` 是对话所有权依据。所有对话列表、读取、修改、删除和发消息查询都同时限定当前登录用户 ID。不能只依赖前端隐藏；Worker 必须逐条验证所有权。

消息保存角色、正文、状态和时间。用户消息先保存，再调用模型；调用失败时保存清晰的失败状态，允许用户重新发送，不伪造助手回复。

删除对话时，同时删除 D1 消息记录和对应 R2 附件。数据库约束和索引覆盖用户、对话、消息时间线及活动会话查询。

## AI 接口配置

设置页保留一个共享 Base URL 和两个独立 Key：

- Base URL 默认 `https://llmhub.ltd/v1`
- 文字模型默认 `gpt-5.5`
- 图片模型默认 `gpt-image-2`
- 文字 API Key：生成笔记、GPT 对话、账号分析、同行分析
- 图片 API Key：仅封面图片生成

API Key 使用 Worker Secret 中的配置加密密钥，经 AES-GCM 加密后存入 D1。接口永远只返回脱敏值。保存时留空代表保留旧 Key；测试文字接口和测试图片接口继续独立执行、独立显示结果。

浏览器不直接调用中转站，所有调用均经 Worker 完成，避免暴露 Key。请求错误区分：未配置、鉴权失败、模型不可用、上游超时、格式错误和网络错误。

## 生成笔记设计

### 场景策略

提示词先判断决策人为什么寻找活动服务，再决定标题和正文角度。固定场景认知如下：

#### 手作 DIY

- 企业：HR 或行政关注员工参与、团队关系、节日关怀和执行省心。
- 商场：运营或市场关注顾客停留、现场互动、打卡内容和商场好感。
- 楼盘：营销或活动负责人关注到访体验、家庭参与、等待过程和项目印象。
- 社区：社区运营关注邻里参与、亲子互动、活动秩序和落地难度。
- 4S 店：市场或门店负责人关注客户等待时间、家庭陪伴体验和品牌好感。

#### Photobooth

- 婚礼：新人关注宾客参与、现场氛围、纪念感和流程配合。
- 企业活动：HR、行政或市场关注品牌呈现、员工参与和现场执行。
- 宝宝宴：家庭关注亲友互动、纪念内容和不同年龄宾客的参与感。
- 聚会：组织者关注快速参与、现场气氛和可带走的纪念内容。

输入事实不足时不得补写城市、人数、客户评价、成交、客流、停留时长和现场反应。

### 两阶段生成

生成继续使用两次文字模型调用，以质量优先：

1. 策略初稿：内部形成受众、客户动机、点击理由和内容角度，再生成 3 个标题与 1 篇正文。
2. 终审重写：依据输入事实逐项核对，删除编造、业务混写、空泛营销词、AI 写作痕迹和违规导流词，最后返回可发布内容。

策略判断和检查结果用于后台质量控制，不在结果区堆叠额外说明。用户最终只看到：

- 3 个标题。
- 标记出的推荐标题。
- 1 篇完整正文，话题已直接附在末尾。

### 标题规则

- 严格 3 个，角度彼此不同。
- 让对应场景的决策人一眼看见与自己有关的问题或结果。
- 优先使用场景、活动形式、真实执行要点和具体顾虑，避免抽象口号。
- 不使用无法证明的“爆单”“翻倍”“全场都在排队”等承诺。
- 不使用低质悬念、夸张标点和与正文无关的点击诱饵。

### 正文规则

- 站在客户决策过程写，不以“我们很专业”为主线。
- 回答客户为什么需要这类活动、现场最担心什么、方案如何配合目标。
- 只使用用户输入和业务固定能力范围内的事实。
- 长短句自然变化，删除 AI 高频词、机械三段式、否定式排比、空泛升华和宣传腔。
- 话题 8 至 12 个，直接放在正文末尾，便于一次复制。
- DIY 与 Photobooth 能力严格隔离。

### 终审清单

在返回前检查：目标受众明确、点击理由成立、客户收益具体、事实有输入依据、标题正文一致、没有业务混写、没有 AI 腔、没有导流违规词、标题数量为 3、正文数量为 1、话题位于正文末尾。任一硬性项不满足时继续修正，而不是直接展示初稿。

## GPT 对话设计

### 页面

桌面端采用左侧对话列表、右侧消息区；移动端对话列表收起为抽屉。页面提供：

- 新建对话。
- 对话自动命名，默认取首条文字的精简内容。
- 手动重命名。
- 删除并二次确认。
- 自动保存消息。
- Markdown 文本显示和代码块复制。
- 图片预览、移除和发送。
- 发送中状态、停止当前请求和明确错误提示。

不提供共享、导出、文件夹、置顶和模型切换。

### 上下文

每次发送时从 D1 读取当前用户、当前对话的有效消息，按照中转站兼容格式调用文字模型。仅发送设定范围内的最近上下文；超过限制时优先保留系统指令、最近消息和必要的历史摘要，防止请求无限增长。

如果中转站支持流式响应，前端逐步显示并在完成后保存；若不支持，自动退回完整响应。中途失败不保存虚假的完成消息。

### 图片

聊天单条消息最多上传 4 张 JPEG、PNG 或 WebP，每张限制 10 MB。Worker 校验 MIME、文件头和大小后写入私有 R2，再将图片内容按中转站支持的多模态消息格式传给 `gpt-5.5`。如果当前模型或中转站不支持图片，页面明确提示，不把失败包装成分析结果。

R2 路径包含用户 ID 和对话 ID。图片读取必须经过已登录 Worker 的所有权校验，不能通过公开桶地址访问。图片随对话删除。

## 封面图制作设计

页面只保留：

1. 一个必填的多行提示词文本框，用户自行粘贴或编辑提示词。
2. 一个参考图片上传区域。
3. 一个“生成图片”按钮。
4. 一个结果图片区域和“保存图片”按钮。

不显示业务类型、场景、案例简介、提示词生成、复制提示词、原图分析、图片评分、封面文字建议、负面提示词、推荐图片或生成历史。

点击生成后只调用图片模型和图片 API Key。提示词及参考图片原样作为生成输入；系统只附加必要的安全和文件格式约束，不自行改写用户提示词。未填写提示词或未上传图片时不发送请求。

参考图和结果图写入 R2 临时目录，设置生命周期自动清理，默认保留 24 小时。结果通过鉴权接口显示和保存，页面刷新后不提供封面历史恢复。

## R2 容量控制

- 聊天附件持续保存，直到员工删除所属对话或管理员删除员工。
- 封面输入和输出默认 24 小时自动删除。
- 上传前进行格式、尺寸和数量限制。
- 删除员工时先删除其会话、对话、消息和 R2 对象，再删除账号。
- 设置页显示数据库和对象存储的近似用量，不引入复杂统计报表。

## API 路由

核心路由分组为：

- `/api/auth/*`：登录、退出、当前用户。
- `/api/admin/users/*`：员工管理，仅管理员。
- `/api/settings/ai/*`：接口配置和独立测试，仅管理员。
- `/api/notes/generate`：两阶段笔记生成。
- `/api/chats/*`：对话列表、创建、重命名、删除、消息读取和发送。
- `/api/uploads/chat`：聊天图片上传。
- `/api/covers/generate`：封面图片生成。
- `/api/files/*`：鉴权后的私有图片读取。
- 现有账号分析和同行分析路由迁移到 Workers 运行时，继续使用文字 Key。

所有业务路由先验证会话。管理员路由再验证角色。文件、对话和消息路由必须验证资源所有权。

## 迁移策略

当前项目没有需要保留的多人对话数据，因此 Cloudflare 首次部署创建空 D1。原本保存在本机的加密 API 配置不上传 GitHub，也不自动迁移；部署后由管理员在设置页重新填写文字 Key 和图片 Key。

本地开发继续支持 `.dev.vars`，该文件加入 `.gitignore`。生产秘密使用 Cloudflare Secrets；部署凭证只存入 GitHub Actions Secrets。仓库只提交变量名示例、D1 migration 和部署说明。

## 错误处理

- 未登录返回 401，权限不足或资源不属于当前用户返回 403/404。
- 图片过大、类型错误和数量超限在调用模型前返回可理解提示。
- 中转站鉴权失败明确指出文字 Key 或图片 Key。
- 上游超时不清除用户输入，允许重试。
- 笔记 JSON 格式异常先执行一次受约束修复，仍失败才返回错误。
- 图片生成失败不保留无效结果记录，并清理临时文件。

## 验证与验收

### 自动测试

- 密码查找、哈希验证、会话过期、停用和重置密码。
- 管理员权限与普通员工越权拦截。
- 两名员工之间对话、消息和图片完全隔离。
- D1 对话 CRUD、消息顺序和删除级联。
- R2 文件类型、大小、私有读取和清理。
- 两个 API Key 的加密存取和独立调用路由。
- 9 个场景的受众策略、业务隔离、3 标题、单正文、话题内联和 humanizer 检查。
- 封面接口只使用图片 Key，不触发文字接口或图片分析。
- 中转站模拟成功、鉴权失败、超时、非流式和格式错误。

### 浏览器验收

- 管理员创建两名员工，两人仅输入各自密码登录。
- 两人创建对话并刷新页面，历史仍在且互相不可见。
- GPT 对话可上传图片、连续提问、重命名和删除。
- 生成笔记输出严格为 3 个标题和 1 篇正文，话题已在正文末尾。
- 不同场景生成内容体现不同决策动机，不出现无依据结果承诺。
- 封面页只显示提示词、上传、生成和结果保存。
- 手机与桌面端可正常使用。
- GitHub 主分支提交后，Actions 先完成 D1 migration 再自动部署 Cloudflare，`pages.dev` 地址可通过 HTTPS 访问。

### 完成标准

只有在类型检查、单元/API 测试、生产构建、Wrangler 配置验证、浏览器回归和线上部署冒烟测试全部通过后，才视为完成。自动测试不得调用真实 API Key 或产生模型费用；线上冒烟测试由管理员使用已配置中转站执行一次文字测试、一次笔记生成、一次聊天和一次图片生成。
